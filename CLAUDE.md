## Architecture Overview

### Application Type
Full B2B2C print shop platform with e-commerce, multi-department job workflow, staff management, invoicing (Paystack), push notifications (Firebase FCM), and a mobile API (Sanctum).

### Two Separate Portals

**Customer Portal** (`routes/web.php`, layout `layouts/theme.blade.php`)
- Middleware `customer.portal` ‚Äî allows guests and customers; redirects staff to admin
- Auth at `/login` and `/register`
- Product browsing ‚Üí order creation ‚Üí payment ‚Üí tracking

**Staff/Admin Portal** (`routes/admin.php`, layout `layouts/admin.blade.php`)
- Auth at `/staff/login` and `/staff/register`
- Middleware `admin.permission:{permission}` enforces role-based access at the route level
- 20+ roles defined in `config/printbuka_admin.php` ‚Äî each role maps to an array of permission strings; `super_admin`/`admin`/`IT` use `['*']`
- `User::hasAdminAccess()` and `User::canAdmin($permission)` are the programmatic checks

### Permission / Role System
All roles and their permission sets live in `config/printbuka_admin.php`. Key sections:
- `roles` ‚Äî maps role slug ‚Üí permission array
- `priorities` ‚Äî role priority scores (super_admin = 100, customer = 0)
- `departments` ‚Äî maps role ‚Üí department label
- `workflow_phases` ‚Äî 6-phase job workflow definitions with gates and responsible roles
- `priorities` (job priorities) ‚Äî `üî¥ Urgent`, `üü° Normal`, `üü¢ Low` (these are stored as-is in the DB; do not rename them without a migration)

### Livewire Volt Components
Files prefixed with `‚ö°` in `resources/views/components/` are Livewire Volt single-file components ‚Äî they contain an anonymous PHP class + the Blade template in one file:
```
resources/views/components/product/‚ö°search.blade.php
```
Referenced as `<livewire:product.search />`. The `‚ö°` prefix is intentional; do not rename these files.

### Image Handling
`app/Support/MediaUrl::resolve($path)` is the single entry point for all image URLs. Resolution order:
1. Data URI ‚Üí pass-through
2. Cloudinary resource ID ‚Üí build CDN URL via `CloudinaryUrl`
3. Full URL ‚Üí return as-is (normalise host if needed)
4. Absolute local path ‚Üí check storage disk
5. Relative storage path ‚Üí check if file exists, serve storage URL
6. Public path ‚Üí `asset()`
7. Last resort ‚Üí attempt Cloudinary upload

All product/job images go through Cloudinary. Config: `CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`, `CLOUDINARY_API_SECRET`.

Product image placeholder (for missing/broken images): `public/img/product-placeholder.svg`.

### Site Settings
`app/Support/SiteSettings` loads config rows from the `site_settings` DB table (5-min cache). Access via the `$siteSettings` variable shared to all views by `AppServiceProvider`. Covers contact info, pricing defaults, service prices, maintenance mode.

### Key Config Files
- `config/printbuka_admin.php` ‚Äî roles, permissions, workflow phases, dropdowns for orders
- `config/printbuka_services.php` ‚Äî service catalog (Direct Image, UV-DTF, DTF, Laser Engraving) with pricing
- `config/cloudinary.php` ‚Äî CDN credentials and default folder (`printbuka`)

### Order Workflow (6 Phases)
`Order::status` moves through: `Quote Requested` ‚Üí `Analyzing Job Brief` ‚Üí `Design / Artwork Preparation` ‚Üí `In Production` ‚Üí `Quality Check & Packaging` ‚Üí `Delivery In Progress` ‚Üí `Client Review ‚Äî Satisfactory`

Advancing phases requires an operations manager approval (`requested_next_status` + `approve-forward`). See `config/printbuka_admin.php` under `workflow_phases` for gates and responsible roles.

Job order numbers follow the `PB-YYYY-XXXX` format generated by `app/Support/ReferenceCode`.

### Payments
Paystack is the payment processor. Keys: `PAYSTACK_PUBLIC_KEY`, `PAYSTACK_SECRET_KEY` in `.env`. The existing `PaymentController` handles callbacks. When integrating Paystack on new flows, follow the existing controller pattern.

### Push Notifications
Firebase FCM via `kreait/laravel-firebase`. Staff devices register via `StaffPushSubscription`. Credentials: absolute path in `FIREBASE_CREDENTIALS` env var pointing to `storage/app/firebase-credentials.json`.

### Frontend Stack
- **Tailwind CSS v4** + **DaisyUI v5** ‚Äî use DaisyUI component classes (`btn`, `card`, `badge`, `modal`, etc.) for UI elements
- **Custom design system** in `resources/css/app.css` ‚Äî brand-prefixed classes: `.pb-btn-primary`, `.pb-card`, `.pb-input`, `.pb-table`, `.pb-badge-*`, etc. Use these for admin UI
- **Icons** ‚Äî `blade-ui-kit/blade-heroicons` (Heroicons v2). Syntax: `<x-heroicon-o-icon-name class="w-5 h-5" />` (outline) or `<x-heroicon-s-icon-name />` (solid)
- **Fonts** ‚Äî Inter (body), DM Sans, DM Serif Display via Google Fonts

### Cloudflare Turnstile (CAPTCHA)
Applied via `VerifyTurnstile` middleware on public-facing forms. Keys: `TURNSTILE_SITE_KEY`, `TURNSTILE_SECRET_KEY`.

### Mobile API
Sanctum token auth at `routes/api.php`. Token prefix `pb_`, 12-hour expiration (`SANCTUM_TOKEN_EXPIRATION=43200`).

### Default DB / Cache / Queue
All three default to `database` driver (SQLite locally). `app/Support/SafeCache` wraps cache operations to handle missing tables during migrations gracefully.
